Лабораторная работа:
Задание по курсу языки описания аппаратуры.

Нужно разработать синтезируемое устройство («конвертер») и систему верификации для него.
Устройство имеет два интерфейса: внутренний (внутрисистемный) и внешний. Внутренний интерфейс: APB/OCP-PIO.
Внешний интерфейс:
Вариант 1: последовательно-параллельный интерфейс, состоящий из трех основных фрагментов: адрес, данные, команда (признак операции + сама операция (запись/чтение)) Вариант 2: UART (однобитная линия «устройство 1»- «устройство 2», контроль передачи через старт-стоп биты)
Вариант 3: последовательный интерфейс (1 бит для передачи адрес-данные-команда + 1 бит признак передачи)
+ в случае Rtl slave : запрос-разрешение передачи.

Взаимодействие master-slave:
-	если устройство master оно при необходимости просто передает данные и ждет подтверждения данных со стороны тестбенча;
-	если устройство slave, то при передаче данных оно выставляет по специальной линии запрос на передачу данных, ждет ответа на него («готов к приему») , после чего передает данные.
Если длина посылки по линии «адрес» или по линии «данные» больше ширины шины, данные передаются за несколько тактов.

Устройство должно сконвертировать транзакцию из внутреннего во внешний интерфейс. Устройство содержит несколько регистров(и/или ОЗУ), в которых нужно хранить адрес, команду, данные для чтения или данные для записи. Передача команды выполняется на частоте в N раз меньше, чем частота внутреннего интерфейса, где
N= (номер_варианта >1)? (номер_вариант % 10) : (3);
Хранение данных в ОЗУ предполагает наличие отдельного модуля ОЗУ (входы: адрес, данные для записи, признак операции, признак чтения; выход — прочитанные данные). Внешнее устройство : верификационный несинтезируемый модуль, который реализует функцию в соответствии с вариантом.
Внутри верификационного модуля содержится память на 32 слова данных. Младшие 16 из них, предназначены для хранения данных, поступающих от устройства «конвертер». После каждой записи верификационный модуль выполняет вычисление по заданию. Результаты складывается или в ячейку памяти (для заданий на основе числового ряда) по адресу 31 или в 16 ячеек, по номерам от 16 до 31.
Синтезируемость устройства подтверждается синтезом в Actel Libero / Altera Quartus / Xilinx Vivado / Xilinx ISE / MG Precision. В отчет по л/р прикладывается лог синтеза + результаты синтеза.

Рекомендуется использование генерации транзакций с помощью генератора псевдослучайных чисел и проверки корректности работы устройства в scoreboard.

Процедура работы с устройством:
1.	Проверить статус устройства. Если устройство свободно, продолжить, в противном случае повторить чтение статуса.
2.	Записать адрес обращения в регистр адреса
3.	Записать команду (0-чтение, 1-запись, 2- потоковая запись, 3-потоковое чтение) Потоковая запись/чтение — передается сразу 4 слова данных
4.	Для операции записи — записать нужное количество слов данных
 
5.	Записать в статусный регистр команду «начать выполнение»
6.	Мастер устройство:
1.	формирует транзакцию
2.	проверяет что слейв готов принять данные
3.	передает транзакцию с командой / данными
4.	в случае чтения переходит в режим ожидания до получения «1» на линии «ответ»
5.	при чтении — принимает ответ
7.	Слейв устройство:
1.	выставляет «1» на линии «запрос обращения»
2.	ждет от мастера «1» на линии «готов к приему»
3.	передает данные/команду
4.	при чтении переходит в режим ожидания до получения «1» на линии «ответ»
5.	при чтении принимает ответ
8.	Агент внешнего интерфейса:
1.	получает команду
2.	сохраняет данные в случае записи
1.	выполняет для полученных данных функцию
2.	сохраняет результат работы
3.	при чтении — передает сохраненные данные


Минимальный рекомендуемый состав системы:
DUT – тестируемое устройство
Test – объект класса TEST_CLASS, который генерирует транзакции для передачи через внутренний интерфейс. Опционально: может генерировать транзакции внешнего интерфейса. Классов, реализующих тесты может быть несколько
, в соответствии с планом верификации.
Агент внешнего интерфейса – объект класса agent_ext, реализует логику работы внешнего устройства (системная функция + ОЗУ)
Драйвер внутреннего интерфейса – объект класса drv_int, реализует взаимодействие с устройством на сигнальном уровне в соответствии с внутренним протоколом. Получает транзакции от Test , преобразует их в транзакции внутреннего интерфейса и передает в DUT. Драйвер внешнего интерфейса – объект класса drv_ext, реализует взаимодействие с устройством на сигнальном уровне в соответствии с внешним протоколом. Получает транзакции от устройства , преобразует их из транзакций внешнего интерфейса в транзакции агента внешнего интерфейса и передает их в Агент.
Scoreboard – модуль сбора статистики. Получает копии всех транзакций и ведет подсчет общего количества транзакций, определяет корректность работы, считает правильно/неправильно обработанные транзакции. В конце моделирования выводит результаты.
Транзакция внешнего/внутреннего интерфейса – класс, поля которого описывают данные передаваемые по соответствующему интерфейсу и реализует необходимые служебные методы .
Связь между классами реализуется любым способом на выбор студента: каналы, очереди, mailbox, вызов методов объекта-получателя и т.д.
Описанная выше структура является минимально-рекомендуемой. По желанию студент может реализовать более полную структуру в рамках классической методологии верификации.
